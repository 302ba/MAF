Modular Application Framework Components

Overview

 Top  Next

Modular Application Framework Components (MAF Components) is a suite of components to build a modular application in Delphi 6 to Delphi 2010. The components are especially designed to work besides any existing application from the beginning, which allows to migrate step by step and retain full functionality during the migration process.

 

Visit our website http://www.maf-components.com for more information

 

 

Advantages using MAF Components

The MAF Components are designed to work with both newly created and old monolith applications. This is achieved by the unique design to work side by side with the application and gives the developer the chance to migrate older projects step by step. For new programs it offers from the start a powerful system to create not only a modular application with only a few lines of code, but also a totally dynamic one with the unique DynamicCode engine. This allows to add new code into old functions, remove obsolete code and even exchange old (maybe buggy) code with new one. All that without recompiling the application or the installed modules.

Integrated into the DynamicCode engine is a code protection system, that compiles the DynamicFunctionTable at start-up for every user with his actual rights to execute certain code or access parts of the application. If the current user does not have the right to access parts of the code, it will not be compiled into the DynamicFunctionTable and cannot be executed.

A extensive resource management keeps exchangeable items likes icons, bitmaps etc. outside the application in a single monolith file and allows to replace them with newer versions centralized at any time. It also manages language resource files, which allow your application to display itself in different languages without even restarting the application.

 

Feature matrix

•Module management

•DynamicCode technology to add/remove and exchange code without re-compiling

•Resource management for graphics/files in a monolith file system

•String resource files to create localized version of the application

•SQL resource files to get applications ready to use several database systems without changing code

•Window management

•Centralized code protection through integrated user management

•Connectivity with easy data exchange between modules

•Global variables, that can be saved and loaded with the next program start again

•Database connectivity with different DACs already supported

•DataStorage and easy data stream handling
